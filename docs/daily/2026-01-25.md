## 【概述】

- K8s 原生资源池抽象
  - Namespace + ResourceQuota
    - Namespace 作为租户/业务线/部门的池子
    - ResourceQuota 限制 Pod 的 requests/limits 总和（CPU、内存、GPU、存储等）
  - Node pools / Node groups
    - 用 label 给节点分组（nodepool=core/batch/gpu）
    - 通过 nodeSelector/affinity/taints&tolerations 调度
  - LimitRange
    - 设置 Pod 的默认/最小/最大 requests 和 limits
    - 防止小任务请求过小、大任务不设 limit

- K8s 原生调度机制
  - kube-scheduler
    - 基于 requests + node 资源可用量调度
    - 支持 Pod 优先级与抢占（PriorityClass + preemptionPolicy）
  - Pod Priority & Preemption
    - 高优先级 Pod 可驱逐低优先级 Pod
  - Overprovisioning（过配）
    - 部署低优先级占位 Pod 提前占满节点
    - 高优业务来时抢占这些占位

- K8s 原生能力的局限
  - 缺少队列级配额与公平共享
  - 缺少超卖/超额使用及回收逻辑
  - 无法按时间段自动调整配额
  - 抢占策略较简单

- 高级调度器
  - Volcano（CNCF）
    - 偏大数据/AI 批处理调度
    - 支持队列、PodGroup、Gang 调度、fair-share、抢占、分层队列
  - Kueue（Kubernetes SIGs）
    - Job 级队列与配额管理
    - ClusterQueue/LocalQueue、flavor、弹性扩缩、抢占
  - Apache YuniKorn
    - 源自 YARN 的 Capacity/Fair 调度器
    - 分层队列、最小/最大配额、公平共享、抢占

- 资源池建模方案
  - 方式 A：Namespace/NodePool + ResourceQuota + 高级调度器
    - 每个池子对应 Namespace + 指定 node pool + ResourceQuota
    - 创建 Queue/ClusterQueue/LocalQueue 与业务绑定
  - 方式 B：完全依赖调度器队列作为资源池
    - Volcano/YuniKorn：队列树即资源池树
    - Kueue：ClusterQueue + ResourceFlavor 定义资源池

- 分时段动态资源管理
  - 用高级调度器的队列配额作为可调上限
    - Volcano：队列支持 deserved/guarantee、capability
    - Kueue：ClusterQueue 定义每种 ResourceFlavor 的配额
    - YuniKorn：队列配置 min/max/capacity
  - 上层控制平面定时更新配额
    - CronJob/外部调度服务按时间段调整
    - 调用 K8s API 更新 Queue/ClusterQueue/ResourceQuota
  - 配合 Cluster Autoscaler 动态扩缩节点

- 空闲资源跨池分配/借用
  - 核心思路：超卖 + 可回收配额
    - 每个池有保证资源（guaranteed）和最大可用资源（capability）
    - 空闲时可被其他池借用，不超过各自最大可用
    - 需求上升时可抢回借出的资源
  - 高级调度器能力
    - Volcano：队列有 deserved、capability、reclaimable
    - YuniKorn：min/max 配额，支持弹性配额
    - Kueue：多个 LocalQueue 共享 ClusterQueue

- 高优任务抢占低优
  - Pod 级优先级（K8s 原生）
    - 定义 PriorityClass（critical/high/normal/low/placeholder）
    - 高优 Pending 时驱逐低优先级 Pod
  - 队列/Job 级抢占（高级调度器）
    - Volcano：队列优先级 + Pod 优先级综合抢占
    - Kueue：Admission 阶段选择性驱逐低优 Workload
    - YuniKorn：基于队列/任务优先级 preempt

- K8s 原生能力清单
  - 资源模型：ResourceRequests/Limits、ResourceQuota、LimitRange、QoS
  - 调度抢占：kube-scheduler、Pod Priority & Preemption、NodeSelector/Affinity、Taints & Tolerations
  - 弹性伸缩：HPA、VPA、Cluster Autoscaler、Overprovisioning
  - 稳定性：PodDisruptionBudget（PDB）

- 推荐落地方案
  - 方案一：偏在线服务 + 少量批处理
    - 以原生 K8s 为主，辅助简单 Volcano/Kueue
    - 在线服务用 Namespace + ResourceQuota + PriorityClass
    - 批处理用低 PriorityClass
  - 方案二：多租户多部门强分时配额
    - 采用 Volcano 或 Kueue 作为统一调度器
    - 队列树映射组织结构
    - 自定义控制器按时间段更新队列配额

## 【详细】

### 资源池建模：两种方式

**方式 A：Namespace/NodePool + ResourceQuota + 高级调度器**

每个"池子"对应：

- 一个 Namespace（如 team-online、team-offline、dept-a、dept-b）
- 指定的 node pool（通过 nodeSelector/nodeAffinity/taints&tolerations）
- 一个 ResourceQuota，保证硬上限

配合高级调度器：

- Volcano：创建分层队列，根队列 → 子队列 → 叶子队列
- 每个队列可设置 deserved（应得/保证资源）与 capability（上限）
- 配置 reclaimable（可被回收）

**方式 B：完全依赖调度器队列**

- Volcano/YuniKorn：队列树就是资源池树
- Kueue：ClusterQueue + ResourceFlavor 定义不同规格资源池（CPU 架构、GPU 型号、Spot/On-Demand 等）

### 分时段动态配额实现

K8s 自身没有时间维度的配额能力，需要构建上层控制：

1. **基础层**：用高级调度器的队列配额作为可调上限
2. **控制层**：CronJob/外部调度服务定时更新配额
3. **资源层**：Cluster Autoscaler 动态扩缩节点

示例：09:00–18:00 在线业务池上限提升，离线池下降；其他时间段相反。

### 超卖与回收模型

每个资源池有两个关键值：

- **保证资源（guaranteed/deserved）**：池子必须拥有的资源
- **最大可用资源（capability/capacity）**：池子最多能用的资源

当池 A 只用 10 核（保证 20 核、最大 100 核）时，其他池 B、C 可以借用这 10 核空闲资源。当池 A 需求上升时，可以回收借出的资源。

典型配置：

- 在线池：保证 60%，上限 100%
- 离线池：保证 40%，上限 100%
- 平时都可使用全部 100%，在线紧张时优先回收离线池超出保证部分

### 两层抢占机制

**Pod 级（K8s 原生）**：

- 定义 PriorityClass
- 高优 Pending 时驱逐低优 Pod
- Overprovisioning 用低优先级占位 Pod 预留资源

**队列/Job 级（高级调度器）**：

- Volcano：优先抢占同级队列中超出 deserved 的任务
- Kueue：Admission 阶段选择性驱逐低优 Workload
- 支持按比例抢占、Gang 调度抢占等复杂策略
