# AI Chat 知识点归档格式

## 文件命名

```
/docs/daily/{YYYY-MM-DD}.md
```

例如: `2026-01-25.md`

## 文档结构

文档使用列表层级表示内容结构，**不使用多级标题**（##, ###）。

### 模板

```markdown
# {YYYY-MM-DD} 知识点归档

来源: {平台名称} 分享链接

---

## 【概述】

- 主要知识点一
  - 子知识点 1.1
    - 细节 1.1.1
    - 细节 1.1.2
  - 子知识点 1.2
- 主要知识点二
  - 子知识点 2.1
  - 子知识点 2.2

---

## 【详细】

### 知识点一：{标题}

{详细解释内容，可以包含代码示例、图表说明等}

### 知识点二：{标题}

{详细解释内容}
```

## 格式规范

### 列表层级

- 第一层: 主要知识点
- 第二层: 子知识点
- 第三层: 具体细节

**最多三层**，保持内容简洁。

### 详细部分

- 每个需要详细展开的知识点使用 `###` 标题
- 包含解释、示例、引用等
- 适合技术细节、代码、原理解释

### 示例

```markdown
# 2026-01-25 知识点归档

来源: ChatGLM 分享链接

---

## 【概述】

- 七层负载均衡必须使用代理模式
  - 加密流量需要终止 TLS 才能查看内容
    - TLS 设计目的就是防止中间人查看明文
    - 不终止 TLS 只能看到 SNI 等握手信息
  - TCP 连接是有状态的端到端协议
    - 连接建立后不能"悄悄换掉"一端
    - 介入连接即成为连接的一端
  - HTTP 是请求/响应模式
    - 每条请求可能需要不同路由
    - 需要始终在线才能处理所有请求

- 负载均衡模式分类
  - Full Proxy（全代理）
    - 维护两条独立连接
    - 可检查和修改请求与响应
  - Half Proxy（半代理）
    - 完成建连阶段后数据直连
    - 只做 L4 端口切换、NAT 等
  - Passthrough（透传）
    - 不终止 TCP
    - 完全不知道应用层内容

---

## 【详细】

### Full Proxy vs Half Proxy

Full Proxy（全代理）是七层负载均衡的标准做法：

- 维护两条独立连接：
  - 客户端 ↔ LB（连接 A）
  - LB ↔ 后端（连接 B）
- 双向都由 LB 参与
- 可以检查和修改请求与响应
- HAProxy、Envoy、Nginx、F5 等都采用此模式

Half Proxy（半代理）：

- 客户端先连到代理
- 代理完成建连阶段（握手、路由选择）
- 真正的数据流在客户端和后端之间直接传输
- DSR、流媒体直连方案属于此类

### TLS Termination 的必要性

要在应用层（HTTP/HTTPS）做负载均衡，必须在同一个 LB 上做 TLS 终止：

- 不终止 TLS = 看不到 HTTP 内容 = 不叫 L7 LB
- 终止 TLS 才能解析 URL、Header、Cookie 等
- 这是"看七层内容"和"只做四层转发"在加密场景互斥的根本原因
```
