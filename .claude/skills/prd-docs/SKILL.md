# PRD 文档撰写技能

## 概述

本技能提供完整的软件产品文档撰写模板和指导，包括产品需求文档（PRD）、技术设计文档和测试用例文档。适用于通用软件产品（Web 应用、移动应用、后台系统等）。

## 文档类型

本技能支持三种文档类型：

| 文档类型 | 适用场景 | 输出格式 |
|----------|----------|----------|
| PRD | 产品需求定义、功能规格说明 | Markdown |
| 技术设计 | 架构设计、API 设计、技术选型 | Markdown |
| 测试用例 | 功能测试、接口测试、性能测试 | Markdown |

## 工作流程

### 方式一：从头开始撰写完整文档

当用户要求撰写某功能的完整文档时：

1. **收集需求信息**：向用户了解产品/功能的基本信息
   - 产品/功能名称
   - 目标用户
   - 核心功能点
   - 特殊需求（性能、安全、合规等）

2. **撰写 PRD**：基于模板生成 PRD 文档
   - 读取 `references/prd-template.md` 作为结构参考
   - 根据用户需求填充内容
   - 输出完整的 PRD 文档

3. **撰写技术设计**：基于 PRD 生成技术设计方案
   - 读取 `references/tech-design-template.md` 作为结构参考
   - 分析 PRD 中的功能需求
   - 设计系统架构、数据模型、API 接口
   - 输出完整的技术设计文档

4. **撰写测试用例**：基于 PRD 和技术设计生成测试用例
   - 读取 `references/test-case-template.md` 作为结构参考
   - 分析功能需求和技术设计
   - 设计功能测试、接口测试、性能测试用例
   - 输出完整的测试用例文档

### 方式二：基于现有 PRD 生成其他文档

当用户已提供 PRD，需要生成技术设计或测试用例时：

1. **阅读现有 PRD**：仔细阅读用户提供的 PRD 文档
2. **生成技术设计**：基于 PRD 内容撰写技术设计文档
3. **生成测试用例**：基于 PRD 和技术设计撰写测试用例文档

### 方式三：单独撰写某种文档

当用户只需要某一种文档时，直接按对应模板撰写。

## 文档结构

### PRD 文档结构

```
1. 文档信息（版本、作者、状态）
2. 文档修订历史
3. 产品概述（背景、定位、目标）
4. 功能范围（功能总览、MVP 范围）
5. 用户角色
6. 详细功能需求（用户故事、业务规则、交互流程）
7. 数据需求（埋点、指标）
8. 非功能需求（性能、安全、兼容性）
9. 里程碑与交付计划
10. 风险与依赖
11. 附录（术语表、参考文档）
```

### 技术设计文档结构

```
1. 文档信息（关联 PRD）
2. 文档修订历史
3. 背景与目标（需求概述、技术目标、约束条件）
4. 系统架构（整体架构、架构说明）
5. 核心模块设计（模块职责、核心流程、时序图）
6. 数据模型设计（实体关系、数据表设计、缓存设计）
7. API 设计（RESTful API、事件/消息设计）
8. 技术选型
9. 关键技术决策
10. 安全设计
11. 监控与运维
12. 测试策略
13. 附录
```

### 测试用例文档结构

```
1. 文档信息（关联 PRD/技术设计）
2. 文档修订历史
3. 测试概述（范围、策略、环境）
4. 功能测试用例（按模块组织）
5. 接口测试用例
6. 性能测试用例
7. 兼容性测试用例
8. 安全测试用例
9. 回归测试清单
10. 测试执行计划
11. Bug 记录
12. 附录
```

## 使用指南

### 开始撰写文档前

通过以下问题收集必要信息：

1. **产品/功能名称**：要设计的产品或功能叫什么？
2. **目标用户**：谁会使用这个产品/功能？
3. **核心功能**：这个产品/功能要解决什么问题？核心能力是什么？
4. **特殊要求**：是否有性能、安全、合规等方面的特殊要求？
5. **技术栈偏好**：是否有技术栈偏好或约束？
6. **时间/资源约束**：是否有开发周期或资源限制？

### 撰写过程中的原则

- **中文撰写**：所有文档内容使用中文
- **Markdown 格式**：输出标准 Markdown 格式
- **结构完整**：按照模板结构，确保章节完整
- **内容具体**：避免空泛描述，给出具体示例
- **可执行性**：文档应具有可执行性，便于后续开发和测试

### 输出方式

- 如果用户指定了文件路径，写入到指定文件
- 如果用户没有指定，询问是否需要写入文件
- 文件名建议格式：`[产品名称]-PRD.md` / `[产品名称]-技术设计.md` / `[产品名称]-测试用例.md`
